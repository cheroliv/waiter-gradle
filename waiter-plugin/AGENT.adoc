== Projet

**Nom** : waiter-gradle — plugin Gradle d'attente/barriere pour pipeline CI/CD
**Stack** : Kotlin, Gradle (plugin development), JUnit 5, functionalTest
**DAG** : N2 — Plugin Gradle (publiable, consommable par engine N3)
**Statut** : Bootstrap — EPIC-0 Onboarding, 8 ateliers CADRAGE_METHODE termines

=== Objectif

Plugin Gradle pour les workflows CI/CD :
- Attente conditionnelle (wait-for API, wait-for service, health check)
- Barrieres de pipeline (gate avant deploy, smoke test)
- Timeout et retry configurables
- Integration native avec GitHub Actions

== Regles Absolues

=== Git

**INTERDICTION FORMELLE** de commit/push/merge sans permission explicite de l'utilisateur.

Commandes autorisees (lecture seule) : `git status`, `git diff`, `git log`, `git show`
Commandes interdites (sauf ordre) : `git add`, `git commit`, `git push`, `git merge`, `git rebase`

=== Tests

L'agent NE DOIT JAMAIS lancer de tests (`./gradlew test`, etc.) lors de la procedure de fin de session SANS permission explicite de l'utilisateur.

=== Securite

* Jamais de secrets en dur dans le code
* Jamais de credentials dans les fichiers de config
* Jamais de tokens API commites dans Git
* Utiliser des variables d'environnement avec fallback

=== Secrets et tokens — INTERDICTION DE LOG

**PRINCIPE** : Les tokens API, mots de passe, et identifiants credentials NE DOIVENT JAMAIS apparaitre dans la sortie terminal de l'agent.

**INTERDICTIONS** :

* **NE JAMAIS** passer un token en argument visible d'une commande
* **NE JAMAIS** afficher le contenu d'un fichier contenant des tokens
* **NE JAMAIS** utiliser `echo`, `cat`, ou toute commande qui affiche des tokens dans le terminal

=== Methodologie de Travail

* **1 fichier a la fois** : Modifier un seul fichier avant de passer au suivant
* **Validation immediate** : Compiler apres chaque modification
* **Contexte leger** : Maximum 50k tokens EAGER (~3000 lignes)
* **Regle d'or** : Mieux vaut 5 sessions de 20 minutes qu'une session de 2 heures avec debugging chaos.
* **Hook ouverture** : Checklist `SESSION_CHECKLIST.adoc` obligatoire

== Gestion du Contexte — CRITIQUE

* **Que faire ensuite ?** -> **`PROMPT_REPRISE.adoc`** (mission session en cours)
* **Roadmap globale** -> **`.agents/INDEX.adoc`** (EPICs, portefeuille, statut)
* **Cadrage methodologique** -> **`CADRAGE_METHODE_*.adoc`** (8 ateliers completes)

== Session Procedure

=== Debut de Session

1. Lire `PROMPT_REPRISE.adoc` pour la mission en cours
2. Lire `.agents/INDEX.adoc` pour les regles absolues + roadmap
3. Executer la checklist `SESSION_CHECKLIST.adoc`

=== Fin de Session

**Mots-cles declencheurs** : "nouvelle session", "je quitte", "session terminee"

**Procedure obligatoire en 6 etapes** :

1. **Creer l'archive LAZY** : `.agents/sessions/{N}-{type}-{sujet}.adoc`
2. **Mettre a jour PROMPT_REPRISE.adoc** — Mission N+1 (max 70 lignes)
3. **Mettre a jour SESSIONS_HISTORY.adoc** — Ajouter la session terminee
4. **Mettre a jour INDEX.adoc** — Date, session N+1, roadmap, portefeuille
5. **Mettre a jour TEST_COVERAGE_ANALYSIS.adoc** (si tests modifies)
6. **Mettre a jour COMPLETED_TASKS_ARCHIVE.adoc** : `.agents/archives/COMPLETED_TASKS_ARCHIVE_{mois}.adoc`

**Aucune etape ne peut etre marquee OK si le fichier n'a pas ete effectivement modifie et verifie.**

=== Mecanisme de backup — Rotation

Tous les **10 sessions**, les donnees historiques sont copiees integralement
dans `.agents/backup/` sans transformation. Les fichiers actifs sont tronques
pour ne contenir que les 10 dernieres sessions (sliding window).
