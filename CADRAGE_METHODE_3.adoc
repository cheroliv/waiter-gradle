== 1. Zoning — Structure modulaire du plugin

Le plugin est dans `engine/`. Structure minimale, tool-agnostique.

```
engine/
├── build.gradle.kts                      ← includeBuild() vers N0/N1/N2
├── settings.gradle.kts                   ← pluginManagement + include("waiter-plugin")
├── AGENT.adoc
├── .agents/
│   ├── INDEX.adoc
│   └── SESSION_CHECKLIST.adoc
│
└── waiter-plugin/
    ├── build.gradle.kts                  ← plugin java-gradle-plugin + kotlin-dsl
    └── src/
        ├── main/kotlin/com/cheroliv/engine/
        │   ├── WaiterPlugin.kt              ← entry point, enregistre les tasks
        │   └── tasks/
        │       ├── ShellTask.kt             ← wrapper shell vers chaque plugin N0/N1/N2
        │       └── CiCdTask.kt              ← commit+push → CI/CD GitHub Actions
        │
        └── test/kotlin/com/cheroliv/engine/
            ├── PluginFunctionalTest.kt       ← TestKit : apply plugin → tasks presentes
            └── ShellTaskTest.kt              ← TestKit : execute shell task → stdout


== 2. Storyboard — Flux d'exécution

=== Flow 1 : Bootstrap (one-shot)

[plantuml]
----
@startuml
actor Dev
participant "Terminal" as T
participant "Gradle" as G

Dev -> T : git clone engine/
Dev -> T : ./gradlew tasks
T -> G : invocation
G -> G : scanne waiter-plugin + includeBuild()
G --> T : liste des taches disponibles
Dev -> T : ./gradlew {task}
----
@enduml

=== Flow 2 : Exécution locale synchrone

[plantuml]
----
@startuml
actor "Utilisateur/Agent" as U
participant "Shell" as S
participant "Gradle Daemon" as D
participant "Plugin N2" as N2

U -> S : ./gradlew -p engine {task}
S -> D : hot-start (daemon deja lance, < 2s)
D -> N2 : execute la tache demandee
N2 --> D : SUCCESS (resultat produit)
D --> S : BUILD SUCCESSFUL + stdout
S --> U : resultat
----
@enduml

=== Flow 3 : Exécution CI/CD (canal principal)

[plantuml]
----
@startuml
actor "Utilisateur/Agent" as U
participant "Engine local" as E
participant "Git" as G
participant "GitHub Actions" as CI
participant "Runner" as R

U -> E : ./gradlew -p engine {task} --mode=ci
E -> G : git commit intention → push
G -> CI : push declenche workflow
CI -> R : ./gradlew {task} (CPU/GPU/RAM du runner)
R -> R : execution (peut durer minutes ou heures)
R -> G : git commit resultat → push
U -> E : git pull → resultat disponible
----
@enduml

=== Consommateurs possibles

Le plugin est tool-agnostique. Tout agent capable d'executer `./gradlew` ou `git pull` peut le consommer :

|===
| Type | Exemple | Comment
| Terminal humain | bash, zsh, fish | `./gradlew -p engine {task}`
| Agent LLM (LangChain4j) | open-webui, opencode, script custom | `Runtime.exec("./gradlew -p engine {task}")`
| Agent LLM (Python) | script Python avec subprocess | `subprocess.run(["./gradlew", "-p", "engine", task])`
| CI/CD | GitHub Actions, Jenkins, GitLab CI | `./gradlew -p engine {task}`
|===
